# QueueStorm Investigator

Support-ticket investigation service for the SUST CSE Carnival 2026 · Codex Community Hackathon — Online Preliminary.

Receives a customer complaint + transaction history, investigates what is actually true, routes the case, and drafts a safe bilingual reply.

---

## Quick Start

```bash
# 1. Clone and enter the repo
git clone <repo-url>
cd preli_app

# 2. Create and activate virtual environment
python3 -m venv .venv
source .venv/bin/activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Configure secrets
cp .env.example .env
# Edit .env and fill in your GROQ_API_KEY (comma-separated for key rotation)

# 5. Start the server
uvicorn app.main:app --host 0.0.0.0 --port 8000

# 6. Verify
curl http://localhost:8000/health
# → {"status":"ok"}
```

### Docker

```bash
docker build -t queuestorm .
docker run -p 8000:8000 --env-file .env queuestorm
```

## Endpoints

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Returns `{"status":"ok"}`. Instant, no dependencies. |
| `POST` | `/analyze-ticket` | Investigate a ticket. Returns structured JSON. |

### Request body (`POST /analyze-ticket`)

```json
{
  "ticket_id": "TKT-001",
  "complaint": "I sent 5000 taka to the wrong number.",
  "language": "en",
  "channel": "in_app_chat",
  "user_type": "customer",
  "transaction_history": [
    {
      "transaction_id": "TXN-9101",
      "timestamp": "2026-04-14T14:08:22Z",
      "type": "transfer",
      "amount": 5000,
      "counterparty": "+8801719876543",
      "status": "completed"
    }
  ]
}
```

`ticket_id` and `complaint` are required. All other fields are optional.

### Response body

```json
{
  "ticket_id": "TKT-001",
  "relevant_transaction_id": "TXN-9101",
  "evidence_verdict": "consistent",
  "case_type": "wrong_transfer",
  "severity": "high",
  "department": "dispute_resolution",
  "agent_summary": "...",
  "recommended_next_action": "...",
  "customer_reply": "...",
  "human_review_required": true,
  "confidence": 0.9,
  "reason_codes": ["wrong_transfer", "transaction_match", "dispute_initiated"]
}
```

### HTTP status codes

| Code | When |
|---|---|
| 200 | Valid analysis. |
| 400 | Malformed JSON or missing `ticket_id` / `complaint`. |
| 422 | Complaint is present but empty after stripping whitespace. |
| 500 | Unexpected internal error. Generic message only — no stack traces or secrets. |

## Tech Stack

- **Runtime:** Python 3.11, FastAPI, Pydantic v2, Uvicorn
- **LLM:** Groq (see MODELS section below)
- **Fallback:** Deterministic rule engine (runs when all LLM keys fail/timeout)
- **Safety:** Two-pass gate enforced in code on every response

---

## AI Approach

### Architecture: Hybrid (LLM judgment + code derivation)

```
request
  → validate (Pydantic; 400/422 on bad input)
  → normalize (Bangla digits → ASCII, phone formats)
  → LLM JUDGMENT (Groq, 20 s timeout, temperature=0)
       Outputs: relevant_transaction_id, evidence_verdict, case_type,
                agent_summary, recommended_next_action, customer_reply, reason_codes
       Uses chain-of-thought: "reasoning" field emitted first so the model
       reasons through the evidence before committing to a verdict.
  → DERIVE (pure code, app/derive.py)
       severity, department, human_review_required, confidence
       — table-driven, sample-calibrated, never generated by the LLM
  → SAFETY GATE (app/safety_gate.py) — rewrites unsafe prose
  → SCHEMA GATE (app/safety_gate.py) — snaps enums, nulls invented IDs
  → 200

  If LLM times out or all keys exhausted:
  → DETERMINISTIC FALLBACK (app/deterministic.py)
       Same pipeline steps using rule-based matching + bilingual templates
```

**Why hybrid?** The LLM is good at reading context and generating natural prose in the right language. But it hallucinates transaction IDs, does arithmetic poorly, and is non-reproducible — so severity, department, human_review_required, and confidence are derived in code from lookup tables calibrated against the 10 public samples. The rules-only path scores well on its own.

### Chain-of-thought reasoning

The LLM is instructed to emit a `reasoning` field *first* in its JSON output, working through: which transaction matches the amount, how many transfers exist to the same counterparty, whether the status matches the complaint, and only then committing to the verdict. The reasoning field is stripped before the response is returned.

### Transaction matching

1. **Duplicate check first** — two completed payments, same amount + counterparty close in time → `duplicate_payment`, `relevant_transaction_id` = the later one.
2. **Amount is the primary key** — parse the amount from the complaint (normalizing Bangla numerals first), match exactly against history.
3. **Disambiguate by counterparty** if multiple amounts match; otherwise → `null` + `insufficient_data`.
4. **Time is secondary** — treated as literal wall-clock hour ±2h, never UTC-converted.
5. **Never pick by recency** on an ambiguous multi-match.

### Evidence verdict

- `consistent` — matched transaction status supports the complaint (failed + "balance deducted"; pending + "not received"; completed + wrong-transfer claim; duplicate pair + "charged twice").
- `inconsistent` — data contradicts the claim: status mismatch, OR behavioural mismatch (≥2 prior transfers to same counterparty contradicts a "wrong number" claim).
- `insufficient_data` — vague complaint, no amount, ambiguous multi-match, or empty history.

## Safety Logic

All safety enforcement runs **in code** on every response, after the LLM or deterministic engine, before the response is returned.

| Rule | Enforcement |
|---|---|
| Never request PIN / OTP / password | `_strip_credential_requests()` drops any sentence asking for credentials |
| Never promise a refund / reversal | `_rewrite_promises()` replaces unauthorized promises with "any eligible amount will be returned through official channels" |
| PIN/OTP reminder for customers | Added automatically for `user_type=customer/unknown`; omitted for `merchant/agent` |
| Customer reply language | LLM matches complaint language; deterministic templates are bilingual (en + bn) |
| Prompt injection | `contains_injection()` detects embedded instructions; forces `human_review_required=True` + reason code; LLM is told to ignore them |
| Invented transaction IDs | `enforce_schema()` nulls any `relevant_transaction_id` not present in the request history |

## MODELS

| Model | Provider | Used for | Why |
|---|---|---|---|
| `llama-3.3-70b-versatile` | [Groq](https://groq.com) | Primary LLM judgment + prose | Fast inference (sub-2s p50), strong reasoning, free tier, JSON-mode support, temperature=0 reproducibility |

**Key rotation:** `GROQ_API_KEY` accepts a comma-separated list (e.g. `key1,key2,key3`). On any failure (rate limit, auth error, timeout) the engine rotates to the next key. If all keys fail, the deterministic fallback answers.

**Fallback model:** None — the deterministic engine uses rule-based matching and bilingual templates, requiring no model at all.

## Assumptions & Known Limitations

- **Time zone:** The service treats transaction timestamps as literal wall-clock time without UTC→Asia/Dhaka conversion. This matches the sample calibration (SAMPLE-01 uses `14:08Z` as "around 2pm").
- **Amount threshold:** The `>= 50000 BDT` high-value guardrail is a heuristic not validated against any sample — kept conservative to avoid fighting the sample-calibrated rules at typical amounts.
- **Groq availability:** If Groq is down or quota is exhausted on all keys during judging, the deterministic fallback handles all cases without any network calls. The deterministic path passes all 10 public samples.
- **Language detection:** Mixed-script detection is heuristic (Unicode range counts). Edge cases between `en` and `mixed` are handled gracefully — the safety gate adds the appropriate reminder in both cases.
